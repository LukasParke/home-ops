# Plan: Add LiteLLM Proxy to the Cluster

## 1. Goal & Scope

Deploy [LiteLLM Proxy](https://docs.litellm.ai/docs/proxy/deploy) into the `default` namespace to provide a single OpenAI-compatible API endpoint that routes to multiple upstream LLM providers (OpenAI, Anthropic, Azure, Bedrock, local Ollama, etc.).

Scope for this iteration:
- LiteLLM proxy + built-in admin UI in one deployment.
- Postgres backend for the UI/usage metadata (cloudnative-pg).
- External HTTPS access via Envoy Gateway (`litellm.${SECRET_DOMAIN}`).
- Flux-managed, SOPS-encrypted secrets, and monitoring via Prometheus + Gatus.

Out of scope (can be added later):
- Redis caching / semantic caching.
- Separate UI-only or gateway-only componentized deployment.
- SSO / OIDC.

## 2. Deployment Model

The repo has two dominant patterns:
1. `bjw-s/app-template` OCI chart for simple/stateless workloads.
2. Upstream Helm charts for complex operators (kube-prometheus-stack, cloudnative-pg).

**Decision:** Use the **official LiteLLM Helm chart** (`deploy/charts/litellm-helm`) because it handles:
- DB migration Jobs.
- Separate proxy/UI component toggles.
- Master key and DB credential secrets.
- HPA / ServiceAccount.

An `OCIRepository` source will be added in the app directory (consistent with the repo's other charts), pointing to `oci://ghcr.io/berriai/litellm-helm`.

## 4. Required Secrets

Add the following keys to `kubernetes/components/sops/cluster-secrets.sops.yaml` (or an app-specific `secret.sops.yaml`):

| Key | Purpose |
|---|---|
| `LITELLM_MASTER_KEY` | Master API key for the proxy (`sk-...`). |
| `LITELLM_SALT_KEY` | Encryption salt for stored credentials. |
| `LITELLM_UI_USERNAME` | Default admin UI login. |
| `LITELLM_UI_PASSWORD` | Default admin UI password. |
| `OPENAI_API_KEY` | Example upstream provider key (repeat pattern for Anthropic, Azure, etc.). |

Use `sops edit kubernetes/components/sops/cluster-secrets.sops.yaml` to add them so Flux substitution and `valueFrom.secretKeyRef` can pick them up.

## 5. Database

Create `kubernetes/apps/default/litellm/app/cluster-postgres.yaml`:

```yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: litellm-database
spec:
  instances: 2
  storage:
    size: 5Gi
    storageClass: nfs
  bootstrap:
    initdb:
      database: litellm
      owner: litellm
```

The owner password is generated by cloudnative-pg into a secret named `litellm-database-app`. The HelmRelease references that secret directly in the `db` block to build `DATABASE_URL`.

## 6. HelmRelease (outline)

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: litellm
spec:
  interval: 1h
  chart:
    spec:
      chart: litellm-helm
      version: "0.5.0"          # pin and let Renovate bump
      sourceRef:
        kind: HelmRepository
        name: litellm
        namespace: flux-system
  values:
    nameOverride: litellm
    fullnameOverride: litellm
    deploymentAnnotations:
      reloader.stakater.com/auto: "true"
    replicaCount: 1
    image:
      pullPolicy: IfNotPresent
    masterkeySecretName: cluster-secrets
    masterkeySecretKey: LITELLM_MASTER_KEY
    environmentSecrets:
      - cluster-secrets
    db:
      useExisting: true
      deployStandalone: false
      endpoint: litellm-database-rw.default.svc.cluster.local
      database: litellm
      url: postgresql://$(DATABASE_USERNAME):$(DATABASE_PASSWORD)@$(DATABASE_HOST)/$(DATABASE_NAME)
      secret:
        name: litellm-database-app
        usernameKey: username
        passwordKey: password
    postgresql:
      enabled: false
    redis:
      enabled: false
    proxy_config:
      model_list:
        # Placeholder model so the proxy starts. Replace with real providers
        # once their API keys are added to cluster-secrets.sops.yaml.
        - model_name: fake-openai-endpoint
          litellm_params:
            model: openai/fake
            api_key: fake-key
            api_base: https://exampleopenaiendpoint-production.up.railway.app/
      general_settings:
        master_key: os.environ/PROXY_MASTER_KEY
        database_url: os.environ/DATABASE_URL
        proxy_batch_writing: true
        ui_username: os.environ/LITELLM_UI_USERNAME
        ui_password: os.environ/LITELLM_UI_PASSWORD
    service:
      type: ClusterIP
      port: 4000
    ingress:
      enabled: false              # use Gateway API HTTPRoute instead
    serviceMonitor:
      enabled: true
```

> The exact `values` keys must be verified against the pinned chart version. LiteLLM is under active development, so `model_list`, `config`, and component names change frequently.

## 7. Ingress (Envoy Gateway)

Create `kubernetes/apps/default/litellm/app/httproute.yaml`:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: litellm
  annotations:
    gethomepage.dev/enabled: "true"
    gethomepage.dev/name: LiteLLM
    gethomepage.dev/description: LLM proxy and API key management
    gethomepage.dev/group: Tools
    gethomepage.dev/icon: mdi-brain
    gethomepage.dev/href: "https://litellm.${SECRET_DOMAIN}"
spec:
  hostnames: ["litellm.${SECRET_DOMAIN}"]
  parentRefs:
    - name: envoy-external
      namespace: network
      sectionName: https
  rules:
    - backendRefs:
        - name: litellm
          port: 4000
```

External DNS will automatically create the record because the Envoy Gateway is annotated for external-dns.

## 10. Implementation Steps

1. **Research & pin versions:**
   - Check the latest stable LiteLLM chart version and container tag.
   - Review the chart `values.yaml` for the exact config schema.

2. **Add OCIRepository:**
   - Create `kubernetes/apps/default/litellm/app/ocirepository.yaml`.
   - Verify Flux pulls the chart (`flux reconcile source ocirepository litellm -n default`).

3. **Prepare secrets:**
   - Generate a strong `LITELLM_MASTER_KEY` (prefix `sk-`).
   - Generate `LITELLM_SALT_KEY`.
   - Generate `LITELLM_UI_PASSWORD`.
   - Add provider keys (OpenAI, Anthropic, etc.).
   - `sops edit kubernetes/components/sops/cluster-secrets.sops.yaml` and add all keys.

4. **Create Postgres cluster:**
   - Add `cluster-postgres.yaml`.
   - Wait for `litellm-database` primary to be ready.

5. **Create app manifests:**
   - `helmrelease.yaml`, `httproute.yaml`, `kustomization.yaml`, `ks.yaml`.
   - Commit and push.

6. **Validate deployment:**
   - `make reconcile` or `flux reconcile kustomization cluster-apps --with-source`.
   - `kubectl get pods -n default -l app.kubernetes.io/name=litellm`.
   - Check logs for migration success and proxy startup.
   - Verify `curl https://litellm.${SECRET_DOMAIN}/health/liveliness`.

7. **Configure models:**
   - Log in to the UI at `https://litellm.${SECRET_DOMAIN}/ui`.
   - Add teams, keys, and model-specific rate limits as needed.

8. **Update Gatus and Homepage:**
   - Add the health check and dashboard entry.

9. **Clean up:**
   - Remove the `.decrypted~cluster-secrets.sops.yaml` file if it was left behind.
   - Ensure no plaintext secrets are committed.

## 11. Rollback Procedure

- If the HelmRelease fails, run:
  ```sh
  make app-errors APP=litellm
  ```
- To revert a bad chart version:
  ```sh
  flux reconcile helmrelease litellm -n default
  ```
  or revert the commit and reconcile.
- For DB corruption, restore from the cloudnative-pg scheduled backup (if configured) or recreate the cluster and re-run migrations.

## 12. Security Notes

- LiteLLM master key grants full proxy access: keep it in `cluster-secrets.sops.yaml` only.
- Do not commit provider API keys in plaintext; always use SOPS or `valueFrom.secretKeyRef`.
- Consider enabling `proxy_batch_writing` and spend tracking only after validating baseline stability.
- The UI should not be exposed without authentication; the default UI credentials are a starting point—rotate them via the UI after first login.
- If only internal clients need the proxy, remove the `HTTPRoute` and use the in-cluster Service `litellm.default.svc.cluster.local:4000`.

## 13. Open Decisions

Before implementation, confirm:

1. **Which upstream providers?** (OpenAI, Anthropic, Azure, Bedrock, Ollama, etc.)
2. **Use UI?** The UI requires Postgres and extra resources; disable it if only proxy functionality is needed.
3. **Caching?** Redis caching can be added later using the existing in-cluster Redis if one is deployed, or a dedicated Redis instance.
4. **Rate limits / budgets?** LiteLLM supports per-key budgets and rate limits; decide if these should be in GitOps-configured `config.yaml` or managed via the UI.
5. **Chart source:** implemented with the official OCI chart (`ghcr.io/berriai/litellm-helm:1.89.3`).
